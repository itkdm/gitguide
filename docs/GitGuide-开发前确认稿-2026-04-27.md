# GitGuide 开发前确认稿

## 文档目的

本文档用于在正式写代码前，统一 GitGuide MVP 的产品范围、交互边界、课程结构、进度记录方式和基础技术方向，减少后续返工。

---

## 1. 已确认的产品基础

### 1.1 标准品牌表达

标准对外表达：

* **GitGuide｜边点边学 GitHub**

标准 Slogan：

* **GitGuide｜边点边学 GitHub**

---

### 1.2 MVP 主学习路径

第一版核心学习闭环：

1. 首页
2. 选择语言
3. 选择第一个任务
4. 完成引导
5. 获得反馈
6. 查看进度
7. 进入下一课

对应页面流转建议：

`首页 -> 语言选择 -> 学习路径页 -> 任务页面 -> 完成反馈 -> 进度页 -> 下一任务`

---

### 1.3 MVP 首批课程

第一版只做以下 3 节核心课程：

1. 认识仓库页面
2. 下载代码
3. 创建仓库

这 3 节课优先级最高，原因如下：

* 覆盖 GitHub 新手最常见的首次使用场景
* 便于设计任务式引导流程
* 能完整体现 GitGuide 的核心产品价值
* 适合作为后续课程模板

---

### 1.4 双语策略

GitGuide 第一版采用整站双语切换：

* 首页默认语言为英文
* 中文模式：中文标题、中文按钮、中文提示、中文反馈
* 英文模式：英文标题、英文按钮、英文提示、英文反馈

不采用以下方案：

* 中文讲解 + 英文按钮混合模式
* 只翻译部分页面内容

这样做的好处：

* 用户认知负担更低
* 文案体系更统一
* 更适合新手入门

---

### 1.5 进度记录策略

进度记录方案已经确认如下：

* 完全保存在本地
* 按步骤记录
* 支持用户重学本课

第一版不包含：

* 账号登录
* 云端同步
* 多设备同步

---

## 2. 建议直接确认的产品边界

### 2.1 产品形态定义

GitGuide 第一版的产品形态建议明确为：

> **GitGuide 不是 GitHub 模拟器，也不是普通流程课，而是以模拟页面为操作载体的任务式学习产品。**

更具体地说：

* 外层是课程流程
* 内层是模拟操作页面
* 核心目标是教学引导，而不是页面复刻

用户体验上应同时具备以下三层特征：

* 有明确任务目标
* 有接近真实场景的操作界面
* 有系统性的引导、反馈和进度记录

因此，第一版的实现原则建议为：

* 优先保证“课程感”强于“高仿感”
* 优先保证任务引导、步骤状态和反馈逻辑完整
* 页面只需要做到“足够像 GitHub 使用场景”，不必追求高度还原
* 在关键节点保留用户主动选择，而不是所有操作都走强制线性流程

不建议走的两个极端：

* 纯流程课：用户只是在跟着步骤点，难以形成对真实 GitHub 页面结构的理解
* 纯页面模拟：用户缺少明确引导，容易迷失，初学者学习成本较高

建议最终统一成一句内部定义：

> **GitGuide 是一个任务式、流程化的 GitHub 学习产品，通过模拟 GitHub 场景页面承载学习任务，并用引导语、反馈和进度系统完成教学闭环。**

---

### 2.2 模拟交互边界

建议将 GitGuide 的模拟交互边界明确为：

> **信息架构和核心操作逻辑参考 GitHub，但视觉设计、品牌表达、页面风格、文案体系和交互细节不直接复刻 GitHub。**

具体建议：

* 页面布局可以参考 GitHub 的主要区域结构
* 任务目标和操作路径可以参考 GitHub 常见流程
* 不直接复制 GitHub 的完整视觉风格
* 不使用与 GitHub 过于接近的品牌表达和界面细节
* 教学页面应明确标注“模拟学习页面”

这样做的价值：

* 保留教学上的熟悉感
* 降低后续品牌和商业化风险
* 保留 GitGuide 自己的产品辨识度

建议在页面底部或显著位置统一放置声明：

* 本页面为 GitGuide 提供的教学模拟页面，并非 GitHub 官方页面。

---

## 3. 三节课的课程拆解

以下拆解以“可直接指导产品、设计、前端实现”为目标。

### 3.1 课程一：认识仓库页面

课程目标：

* 让用户理解 GitHub 仓库页面的基本结构
* 认识仓库名、README、文件列表、导航区、Code 按钮等核心区域

建议步骤：

1. 识别仓库标题区域
2. 找到仓库主导航
3. 找到 README 区域
4. 找到文件列表区域
5. 找到 Code 按钮
6. 完成页面结构认知

步骤定义建议：

| 步骤 | 用户动作 | 成功判定 | 系统反馈 |
| --- | --- | --- | --- |
| 识别仓库标题 | 点击仓库标题区域 | 点击目标区域成功 | 提示“这是仓库名称区域” |
| 找到主导航 | 点击导航 Tab 区域 | 点击 `Code` / `Issues` / `Pull requests` 所在导航成功 | 提示“这里是仓库主导航” |
| 找到 README | 点击 README 模块 | 点击目标区域成功 | 提示“README 用于介绍项目” |
| 找到文件列表 | 点击文件列表模块 | 点击目标区域成功 | 提示“这里展示仓库中的文件和目录” |
| 找到 Code 按钮 | 点击 Code 按钮 | 点击目标按钮成功 | 提示“Code 按钮用于获取仓库代码” |
| 完成认知 | 点击完成按钮或自动进入完成态 | 前置步骤均完成 | 展示完成反馈 |

本课完成条件：

* 必须完成全部核心区域识别

本课不需要的复杂行为：

* 不需要真实切换 Tab 内容
* 不需要真实加载仓库数据
* 不需要模拟深层页面跳转

---

### 3.2 课程二：下载代码

课程目标：

* 让用户理解获取代码的两种常见方式
* 学会通过 `Download ZIP` 下载代码
* 学会理解 `git clone` 的基本用途

本课采用多分支完成策略。

#### 路径 A：下载 ZIP

建议步骤：

1. 点击 Code 按钮
2. 查看下载菜单
3. 找到 Download ZIP
4. 点击 Download ZIP
5. 完成 ZIP 下载学习

#### 路径 B：Git Clone

建议步骤：

1. 点击 Code 按钮
2. 查看克隆地址区域
3. 选择 HTTPS 或默认克隆方式
4. 点击复制地址
5. 理解 `git clone` 用法
6. 完成 Clone 学习

步骤定义建议：

| 步骤 | 用户动作 | 成功判定 | 系统反馈 |
| --- | --- | --- | --- |
| 打开 Code 菜单 | 点击 Code 按钮 | 菜单成功展开 | 提示“这里可以获取代码” |
| 选择路径 | 点击 ZIP 或 Clone 入口 | 进入任一路径 | 提示对应说明 |
| 路径 A：下载 ZIP | 点击 Download ZIP | 正确点击该操作 | 提示“适合不使用 Git 的初学者” |
| 路径 B：复制地址 | 点击复制 Clone 地址 | 复制动作成功 | 提示“你已经拿到仓库地址” |
| 路径 B：理解命令 | 查看或确认 `git clone <url>` | 点击“我明白了”或完成提示确认 | 提示“Clone 是把远程仓库复制到本地” |
| 完成课程 | 完成任一路径或两条路径 | 满足完成条件 | 展示完成反馈 |

本课完成条件建议：

* 最低完成标准：完成 ZIP 路径或 Clone 路径中的任意一条
* 进阶完成标准：两条路径都完成，显示“额外掌握”

分支呈现方式已确认如下：

* 用户先进入模拟仓库首页
* 点击 `Code` 按钮后，系统给出解释和提示
* 用户在展开的菜单或引导提示中，自主选择后续路径
* 用户可以选择 `Download ZIP` 路径，也可以选择 `Git Clone` 路径
* 系统根据用户实际点击和选择，进入对应分支引导

这意味着本课不是强制线性流程，而是“同一入口下的自主分支学习”。

这样设计的好处：

* 更贴近真实用户需求差异
* 支持任务分支机制
* 为后续“多解法课程”打基础

### 3.3 课程三：创建仓库

课程目标：

* 让用户理解创建仓库时的几个核心字段
* 学会完成一个最基础的仓库创建流程

建议步骤：

1. 进入创建仓库页面
2. 填写仓库名称
3. 填写仓库描述
4. 选择公开或私有
5. 选择是否初始化 README
6. 点击创建仓库
7. 完成创建流程反馈

步骤定义建议：

| 步骤 | 用户动作 | 成功判定 | 系统反馈 |
| --- | --- | --- | --- |
| 进入页面 | 点击“创建仓库”入口 | 成功进入模拟创建页 | 提示“我们来创建第一个仓库” |
| 填写名称 | 输入仓库名称 | 非空且满足基础格式 | 提示“仓库名称是项目的唯一标识之一” |
| 填写描述 | 输入描述 | 可为空或按规则完成 | 提示“描述帮助别人理解项目用途” |
| 选择可见性 | 选择 Public 或 Private | 已选择一个选项 | 提示“可见性决定谁能看到仓库” |
| 初始化 README | 勾选 README 选项 | 可选步骤或建议步骤 | 提示“README 能帮助项目更容易被理解” |
| 点击创建 | 点击创建按钮 | 核心字段满足条件 | 展示创建成功反馈 |
| 完成课程 | 进入成功页 | 创建流程结束 | 更新进度 |

本课完成条件建议：

* 仓库名称必填
* 可见性必选
* 描述可选
* README 勾选可作为建议项，而不是主线必做步骤

## 4. 统一任务模型建议

为了避免每节课都写一套独立逻辑，建议第一版就使用统一任务模型。

### 4.1 Step 基本结构

建议每个步骤至少包含以下字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 步骤唯一标识 |
| `courseId` | 所属课程 |
| `title` | 步骤标题 |
| `description` | 步骤说明 |
| `targetId` | 对应页面中的目标元素标识 |
| `actionType` | 操作类型，如 `click`、`input`、`select`、`confirm` |
| `successRule` | 成功判定规则 |
| `nextStepId` | 下一步骤 |
| `branchNext` | 分支步骤映射 |
| `hint` | 提示内容 |
| `locale` | 双语文案内容 |

---

### 4.2 课程基本结构

建议每门课至少包含以下字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 课程唯一标识 |
| `slug` | 路由用标识 |
| `title` | 课程标题 |
| `summary` | 课程简介 |
| `language` | 文案语言映射 |
| `entryStepId` | 入口步骤 |
| `steps` | 步骤列表 |
| `completionMode` | 完成模式，如 `all` 或 `any-branch` |
| `resettable` | 是否允许重学 |

---

### 4.3 推荐完成模式

建议第一版支持两种完成模式：

* `all`
  说明：必须完成全部步骤
  适用：认识仓库页面、创建仓库

* `any-branch`
  说明：完成任意一个有效分支即可过关
  适用：下载代码

如果后续需要，再扩展：

* `all-branches`
* `optional-steps`
* `bonus-steps`

---

## 5. 本地进度记录建议

### 5.1 建议记录粒度

建议按 step 存储，而不是只存“学到第几步”。

原因：

* 兼容分支任务
* 兼容可选步骤
* 兼容“重学本课但保留历史完成记录”的后续扩展

---

### 5.2 建议存储内容

建议至少记录以下内容：

| 字段 | 说明 |
| --- | --- |
| `selectedLanguage` | 当前语言 |
| `startedCourses` | 已开始课程 |
| `completedCourses` | 已完成课程 |
| `stepStatus` | 每个步骤的完成状态 |
| `branchChoice` | 分支课程中用户选择了哪条路径 |
| `lastVisitedCourse` | 最近学习的课程 |
| `lastVisitedStep` | 最近学习的步骤 |

状态示例：

* `not_started`
* `in_progress`
* `completed`

---

### 5.3 重学本课建议

建议提供“重学本课”按钮，并采用以下规则：

* 清空该课程的步骤状态
* 保留全局语言设置
* 可选择是否保留课程已完成徽标

第一版建议：

* 点击重学本课后，直接清空该课程步骤状态
* 重新进入该课程的第一个步骤

## 7. 设计与合规建议

### 7.1 设计方向

建议采用以下原则：

* 保留“任务式教学”的清晰结构
* 不做 GitHub 风格复刻
* 保持界面友好、轻量、清楚
* 用自己的品牌色、插画风格、按钮系统和提示样式

建议避免：

* 与 GitHub 过于接近的整体页面配色
* 直接复制 GitHub 导航、图标和布局比例
* 让用户误以为这是 GitHub 官方页面

---

### 7.2 合规建议

建议统一加入以下表达：

中文：

> 本页面为 GitGuide 提供的教学模拟页面，用于帮助用户学习 GitHub 的基础操作，不代表 GitHub 官方产品。

英文：

> This page is a teaching simulation provided by GitGuide to help users learn basic GitHub workflows. It is not an official GitHub product.

---

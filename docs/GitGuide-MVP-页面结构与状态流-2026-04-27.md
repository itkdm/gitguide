# GitGuide MVP 页面结构与状态流

## 文档目的

本文档用于明确 GitGuide MVP 的页面结构、路由建议、核心状态、课程推进方式和本地数据流，作为设计和前端开发的直接参考。

---

## 1. 产品形态落地原则

基于前面的确认，GitGuide MVP 应按以下原则落地：

* 它首先是一个任务式学习产品
* 它通过模拟页面承载操作练习
* 它不是 GitHub 页面复刻项目
* 它必须具备清晰的步骤状态和反馈闭环

因此，页面设计上要同时满足两件事：

1. 用户能感受到“我正在一个类似 GitHub 的场景里操作”
2. 用户能始终知道“我当前在学什么、下一步做什么、完成后会发生什么”

---

## 2. 页面总览

MVP 建议包含以下页面：

1. 首页
2. 学习路径页
3. 课程页：认识仓库页面
4. 课程页：下载代码
5. 课程页：创建仓库
6. 课程完成反馈页
7. 学习进度页

说明：

* 首页默认语言为英文
* 语言切换可以在首页和所有课程页可见
* 第一版不单独拆“语言选择页”，而是在首页提供明确切换入口

这样可以减少无效跳转，并保留“默认英文”的产品设定。

---

## 3. 路由建议

建议采用清晰、稳定、可扩展的路由结构：

| 页面 | 建议路由 |
| --- | --- |
| 首页 | `/` |
| 学习路径页 | `/learn` |
| 课程一 | `/learn/repository-basics` |
| 课程二 | `/learn/download-code` |
| 课程三 | `/learn/create-repository` |
| 课程完成反馈页 | `/learn/:courseSlug/complete` |
| 学习进度页 | `/progress` |

如需支持语言前缀，后续可以扩展为：

* `/en/...`
* `/zh/...`

但第一版更建议将语言状态保存在本地，而不是直接放进路由。

原因：

* 减少首版实现复杂度
* 更适合本地进度记录
* 页面切换时不需要维护双套 URL

## 4. 信息架构

### 4.1 首页

首页目标：

* 快速说明 GitGuide 是什么
* 让用户开始第一节课
* 提供语言切换入口
* 提供继续学习入口

首页建议模块：

1. 顶部品牌区
2. 一句话介绍
3. 主 CTA
4. 语言切换器
5. 三节课程预览
6. 继续学习入口
7. 合规说明

首页按钮建议：

* `Start Learning`
* `Continue`
* `Switch to 中文`

首页状态逻辑：

* 首次进入：主按钮显示 `Start Learning`
* 已有学习记录：显示 `Continue`
* 已完成全部课程：显示 `Review Progress`

---

### 4.2 学习路径页

学习路径页目标：

* 展示 3 节课的顺序
* 展示每节课当前状态
* 让用户进入课程

页面建议模块：

1. 顶部导航
2. 路径标题和说明
3. 课程卡片列表
4. 当前进度摘要
5. 返回首页 / 查看进度入口

课程卡片建议展示：

* 课程名称
* 课程简介
* 状态标签
* 预计学习时长
* 进入按钮

课程状态建议：

* `Not Started`
* `In Progress`
* `Completed`

---

### 4.3 课程页通用结构

三节课虽然页面内容不同，但建议共享统一壳结构。

课程页通用布局建议：

1. 顶部导航栏
2. 页面标题区
3. 模拟场景区
4. 引导层
5. 右侧或底部任务面板
6. 底部说明区

课程页核心区域说明：

* 顶部导航栏：品牌、语言切换、返回学习路径、查看进度
* 页面标题区：课程标题、当前步骤、完成进度
* 模拟场景区：承载类 GitHub 页面结构
* 引导层：高亮、气泡、遮罩、箭头、提示
* 任务面板：当前任务说明、按钮说明、步骤提示、反馈
* 底部说明区：合规声明、术语解释入口

---

### 4.4 课程完成反馈页

课程完成反馈页目标：

* 强化完成感
* 给出本课收获总结
* 引导进入下一课

页面建议内容：

1. 完成状态标题
2. 你学会了什么
3. 本课关键术语
4. 下一课按钮
5. 返回学习路径按钮
6. 查看学习进度按钮

对 `下载代码` 课程建议额外展示：

* 用户完成的是 ZIP 路径、Clone 路径，还是两条都完成

---

### 4.5 学习进度页

学习进度页目标：

* 让用户明确看到自己已完成内容
* 支持继续学习
* 支持重学单课

页面建议内容：

1. 总进度概览
2. 三节课程进度卡片
3. 最近学习记录
4. 重学本课按钮
5. 返回学习路径按钮

每节课卡片建议展示：

* 课程标题
* 完成百分比
* 已完成步骤数
* 当前状态
* 继续学习 / 重学本课

---

## 5. 页面状态流

### 5.1 首次用户状态流

首次进入的默认流程建议如下：

`首页 -> 点击 Start Learning -> 学习路径页 -> 进入第一课 -> 完成课程 -> 课程完成反馈页 -> 返回学习路径 -> 第二课`

关键特点：

* 首页默认英文
* 用户可以在任意时刻切换中文
* 第一节课开始前不强制弹独立语言选择页

---

### 5.2 回访用户状态流

有本地进度记录时，建议流程如下：

`首页 -> 点击 Continue -> 最近学习课程 -> 回到上次步骤`

如果最近课程已完成，则：

`首页 -> Continue -> 学习路径页`

---

### 5.3 课程内状态流

通用课程内状态：

1. `not_started`
2. `in_progress`
3. `step_completed`
4. `course_completed`

每个步骤建议有以下子状态：

1. `idle`
2. `active`
3. `success`
4. `error_optional`

说明：

* `error_optional` 不是“失败惩罚”，而是“操作未命中目标时给轻提示”
* 第一版不建议做严重失败状态
* 新手学习产品应以鼓励和纠正为主

---

### 5.4 下载代码课程状态流

这是 MVP 中最特殊的一课，因为它有分支。

建议状态流：

`进入课程 -> 点击 Code -> 菜单展开 -> 用户选择 ZIP 或 Clone -> 进入对应路径 -> 完成路径 -> 课程完成反馈`

分支逻辑建议：

* 入口步骤只有一个：点击 `Code`
* 进入菜单后，系统提示用户可以选择不同方式获取代码
* 用户点击哪条路径，就激活哪条分支步骤
* 完成任意路径即可完成主课程
* 如果用户继续完成第二条路径，可解锁“额外掌握”状态

推荐状态字段：

* `selectedBranch`
* `completedBranches`
* `hasBonusCompletion`

---

## 6. 课程页交互层设计

### 6.1 模拟场景层

模拟场景层负责承载类 GitHub 结构。

例如：

* 仓库标题区
* 标签导航区
* README 区域
* 文件列表区域
* `Code` 菜单区
* 创建仓库表单区

这一层的职责是“让用户能操作”，而不是“让用户觉得这是 GitHub 官方页面”。

---

### 6.2 引导层

引导层负责教学。

建议包含：

* 遮罩层
* 高亮框
* 引导气泡
* 当前步骤文案
* 下一步说明

引导层规则建议：

* 当前目标必须有明显高亮
* 提示文案要短
* 一次只强调一个主要动作
* 用户误点时给轻提示，不打断流程

---

### 6.3 任务面板

任务面板是课程感的核心组件。

建议固定显示以下内容：

* 当前任务标题
* 当前任务说明
* 当前步骤编号
* 操作提示
* 完成状态

可选内容：

* 术语解释按钮
* 跳过说明按钮
* 重置本课按钮

第一版不建议加入太多复杂控制项，以免打断学习体验。

---

## 7. 本地状态模型建议

### 7.1 全局状态

建议 MVP 至少维护以下全局状态：

| 状态 | 说明 |
| --- | --- |
| `language` | 当前语言，默认 `en` |
| `courses` | 全部课程配置 |
| `courseProgress` | 各课程学习进度 |
| `lastVisitedCourse` | 最近学习课程 |
| `lastVisitedStep` | 最近学习步骤 |
| `ui` | 当前 UI 状态，例如菜单是否展开 |

---

### 7.2 课程进度状态

建议每节课都维护独立进度对象：

| 字段 | 说明 |
| --- | --- |
| `courseId` | 课程 ID |
| `status` | `not_started` / `in_progress` / `completed` |
| `activeStepId` | 当前步骤 |
| `completedStepIds` | 已完成步骤 |
| `selectedBranch` | 当前分支，仅下载代码课需要 |
| `completedBranches` | 已完成分支列表 |
| `startedAt` | 开始时间 |
| `completedAt` | 完成时间 |

---

### 7.3 UI 状态

课程页还会有一些临时 UI 状态，建议和进度状态分开。

例如：

* `isCodeMenuOpen`
* `highlightTargetId`
* `tooltipVisible`
* `activeTooltipId`
* `showCompletionModal`

这样做的好处：

* 学习进度和临时界面状态解耦
* 刷新恢复时逻辑更清楚
* 更适合后续扩展课程组件

---

## 8. 页面与状态的对应关系

### 8.1 首页

依赖状态：

* `language`
* `courseProgress`
* `lastVisitedCourse`

主要行为：

* 切换语言
* 开始学习
* 继续学习
* 查看进度

---

### 8.2 学习路径页

依赖状态：

* `courses`
* `courseProgress`

主要行为：

* 浏览课程
* 进入课程
* 查看课程状态

---

### 8.3 课程页

依赖状态：

* `language`
* `activeCourse`
* `activeStep`
* `courseProgress`
* `ui`

主要行为：

* 执行当前步骤
* 展开局部菜单
* 进入分支
* 显示成功反馈
* 更新本地进度

---

### 8.4 完成反馈页

依赖状态：

* `activeCourse`
* `courseProgress`
* `language`

主要行为：

* 展示课程完成结果
* 进入下一课
* 返回学习路径

---

### 8.5 进度页

依赖状态：

* `courseProgress`
* `language`

主要行为：

* 展示总进度
* 继续学习
* 重学本课

---

## 9. 核心组件清单

建议第一版优先抽出以下组件：

1. `AppHeader`
2. `LanguageSwitcher`
3. `CourseCard`
4. `ProgressSummary`
5. `SimulationFrame`
6. `TaskPanel`
7. `StepHighlight`
8. `GuideTooltip`
9. `CompletionCard`
10. `RestartCourseButton`

与课程相关的场景组件建议拆分：

* `RepositoryOverviewScene`
* `DownloadCodeScene`
* `CreateRepositoryScene`

这样可以保证：

* 通用学习壳可复用
* 不同课程的页面场景可独立演进

---

## 10. 课程切换与恢复规则

### 10.1 进入课程规则

建议规则：

* 用户可从学习路径页进入任意课程
* 若课程有历史进度，则默认恢复到上次步骤
* 若用户选择重学，则从第一步开始

---

### 10.2 完成课程规则

建议规则：

* 完成课程后写入 `completed`
* 更新 `completedAt`
* 写入最近完成课程信息
* 跳转到课程完成反馈页

---

### 10.3 继续学习规则

建议规则：

* 如果 `lastVisitedCourse` 存在且未完成，则跳回该课
* 如果最近课程已完成，则跳转学习路径页
* 如果全部课程完成，则首页主按钮显示“查看进度”

---

## 11. 第一版不建议做的内容

为了控制复杂度，以下内容建议明确不纳入 MVP：

* 账号系统
* 服务端存储
* 浏览器多标签同步
* 真正的 GitHub API 接入
* 真实代码下载
* 真实仓库创建
* 复杂动画编排
* 高仿 GitHub 全站导航系统

---

## 12. 建议的开发顺序

建议开发顺序如下：

1. 搭建全局页面框架和路由
2. 实现语言状态和本地存储
3. 实现学习路径页和课程卡片状态
4. 搭建课程通用壳结构
5. 实现步骤引导和高亮逻辑
6. 实现课程一：认识仓库页面
7. 实现课程二：下载代码分支流程
8. 实现课程三：创建仓库
9. 实现完成反馈页和进度页
10. 补齐中英文文案

---

## 13. 下一份最适合补的文档

如果继续往工程设计推进，下一份最值得补的文档是：

* `GitGuide 课程配置 Schema 草案`

因为一旦 Schema 定下来，课程内容、步骤逻辑、本地进度结构和页面渲染方式就都可以开始稳定实现。
